# lifeOS Report — 报告生成

生成 lifeOS 各级报告文件（日报/周报/月报/季报），使用标准模板和正确的导航链接。

## 生成脚本

**路径**: `scripts/generate-report.mjs`
**运行**: `node scripts/generate-report.mjs <type> [--date <date>]`

| 类型 | 日期格式 | 默认值 | 示例 |
|------|---------|--------|------|
| `daily` | YYYY-MM-DD | 今天 | `node scripts/generate-report.mjs daily` |
| `weekly` | YYYY-MM-DD | 本周（自动计算周一） | `node scripts/generate-report.mjs weekly` |
| `monthly` | YYYY-MM | 本月 | `node scripts/generate-report.mjs monthly --date 2026-07` |
| `quarterly` | YYYY-QN | 本季 | `node scripts/generate-report.mjs quarterly --date 2026-Q3` |

## 生成行为

- 自动计算：星期、ISO 周数、上下层导航链接、同层连续导航
- 自动检测：如果文件已存在，报错并退出（不覆盖）
- 输出路径：生成后打印文件路径供后续操作

## 输出目录

| 类型 | 目录 | 文件命名 |
|------|------|---------|
| 日报 | `apps/web/content/6-daily/` | `YYYY-MM-DD.md` |
| 周报 | `apps/web/content/5-weekly/` | `YYYY-MM-WNN.md` |
| 月报 | `apps/web/content/4-monthly/` | `YYYY-MM.md` |
| 季报 | `apps/web/content/3-quarterly/` | `YYYY-QN.md` |

## Agent 操作指南

### 当用户要求"创建/新建/写日报/周报/月报/季报"时

**必须**先调用脚本生成文件，然后再编辑填充内容：

```bash
# 1. 生成今天的日报
node scripts/generate-report.mjs daily

# 2. 生成本周周报
node scripts/generate-report.mjs weekly

# 3. 生成指定日期的日报
node scripts/generate-report.mjs daily --date 2026-07-07

# 4. 生成指定月的月报
node scripts/generate-report.mjs monthly --date 2026-08
```

脚本输出示例：
```
已创建: /Users/tangwen/lifeOS/apps/web/content/6-daily/2026-07-06.md
标题: 日报 - 2026-07-06 周一
```

### 生成后的流程

1. 读取生成的文件
2. 根据用户需求填充具体内容（今日计划、任务列表等）
3. 如需关联项目任务树，读取 `public/{project-slug}.tasks.json`
4. 使用 Write 或 SearchReplace 工具更新文件内容

### 模板结构

**日报**包含：今日优先（表格）、时间安排（表格）、规划（今日目标+备注）、进展记录、总结（完成/未完成/收获/明日跟进）、日记

**周报**包含：本周主题、本周核心目标（3条）、本周重点任务（工作/学习/生活/社交四个维度）

**月报**包含：本月主题、月度计划（核心方向+阶段划分）、关键数据表格、月度反思

**季报**包含：本季度核心目标、各维度进展表格、季度反思（做得好/需改进/感悟）、下季度计划

### 注意事项

- 不要手动创建报告文件，始终使用脚本（确保导航链接和 frontmatter 格式正确）
- 如果文件已存在，脚本会报错，此时改用 SearchReplace 编辑现有文件
- 周报的 slug 格式为 `YYYY-MM-WNN`，其中 MM 是该周周一所在的月份

## 报告内容写作规范

### 原则：细节外置、引用内联

报告中出现的「详细方案」/「参考资料」/「背景说明」等长内容，**不写在报告内**，统一放到 [附录](../appendix/) 下。在报告中只用一句话 + 链接引用。

**原因**：
- 同一份方案可能多日/多周需要（比如训练计划、学习路径），避免重复
- 附录是单一事实源（Single Source of Truth），修改一处全报告生效
- 报告保持简洁，重点放在「今天做了什么」

### 章节顺序（日报）

```
今日优先 → 时间安排 → 今日计划任务 → 习惯打卡 → 定期任务
→ 规划（今日目标/备注）→ 进展记录 → 总结（完成/未完成/收获/明日跟进）→ 日记
→ 待安排 ← 放在最底部！
```

**「待安排」必须放最底部**。它是项目里所有未定日期的任务列表（自动从 `projects/*/tasks.json` 抽取），通常很长。把「规划」「进展记录」「总结」「日记」这些需要阅读和填写的内容都推到它前面。

### 跨报告引用约定

| 场景 | 写法 | 指向 |
|------|------|------|
| 详细方案在附录 | `[附录：xxx](../appendix/.../xxx.md)` | 附录具体页面 |
| 相关日报 | `[前一天日报](../6-daily/YYYY-MM-DD.md)` | 同层连续导航 |
| 上层报告 | `[MM 月报](../4-monthly/YYYY-MM.md)` | 月报/季报/年报 |
| 同主题其他日报 | `参见 [YYYY-MM-DD 日报](../6-daily/...md)` | 历史日报 |

**反模式**：在日报内展开大段背景/方案 → 违反「细节外置」原则

### 备注区使用原则

「规划 → 备注」区只放**当天特有的简短补充**（一行以内），长内容必须链接到附录。

**✅ 正确示例**：
```
### 备注

- **引体向上动作不标准**：替代方案及 4-6 周渐进计划见 [附录：引体向上渐进训练方案](../appendix/life/pull-up-progression.md)
```

**❌ 错误示例**：
```
### 备注

- **引体向上渐进训练方案**：今天训练时发现...后续按分阶段推进：①打地基（1-2 周）— 死虫子悬垂 30-60s × 3、肩胛引体 5s × 8-10、澳洲引体 3 × 15；②减重引体（2-4 周）— 弹力带辅助引体 4 × 6-8、离心引体 5-6s 缓慢下落 4 × 3-5、高位下拉 4 × 10-12；③解锁标准（4-6 周）— 半程 → 全程 + 顶部等长保持...
```

### 进展记录 / 总结中的引用

「进展记录」和「总结」里如果需要补充详细方案，使用「见附录：xxx」交叉引用，不要复制内容。

**✅**：`完整替代方案及后续 4-6 周渐进计划见 [附录：引体向上渐进训练方案](../appendix/life/pull-up-progression.md)`
**❌**：`完整替代方案见备注。`（指向同文档内的备注，且备注本身应该已经指向附录了）

### 附录创建规则

新增长内容时，在 `apps/web/content/appendix/{category}/` 下创建独立页面，frontmatter 包含：
- `type: appendix`
- `category`: 与同目录其他文件一致（生活/职业规划/收藏/随笔/其他）
- `slug`: 英文短横线连接
- `tags`: 主题相关

页面末尾建议添加「引用记录」区，列出哪些日报引用了它，便于反向追踪。

### 何时更新附录 vs 新建附录

- **更新现有附录**：方案/参考资料演进（如训练计划进展、工具变化）
- **新建附录**：全新主题，与现有附录无重叠

### 自动显示机制

部分附录数据支持在日报详情页自动显示：
- `apps/web/public/events.json` → 日报页顶部「当日事件」卡片（按 date 匹配）
- `apps/web/public/first-times.json` → 日报页顶部「初体验」卡片（按 date 匹配）

新增类似机制时：建 JSON 数据源 + 在 `ReportDetail.tsx` 加组件 + 在 `ReportPages.tsx` 接入 `showXxxForDate` 参数。

## 报告周期与工作流

### 每日

**规划**（当日或前一天晚上）：
1. 读取当天日报（`content/6-daily/YYYY-MM-DD.md`），不存在则创建（用 `generate-report.mjs daily`）
2. 回顾昨天日报的「明日待跟进」
3. 检查本周计划
4. `node scripts/plans.mjs daily` 查当天待办与已排事件
5. 填今日优先和时间安排

**总结**（晚上）：
1. 提醒回顾完成情况
2. 整理「进展记录」、「总结」（完成/未完成/收获/明日跟进）
3. 未完成的 TODO 迁移到次日
4. 通过 `send-email` 发送「当天总结 + 明天预告」

### 每周（周日晚上）

1. 汇总本周日报要点
2. 写周报：本周关键事项 + 本周完成 + 本周进展 + 下周计划
3. 创建下周的周报和日报文件
4. 通过 `send-email` 发送「本周总结 + 下周预告」

### 每月（月末最后一天）

1. 汇总本月周报
2. 写月报：本月关键事项 + 各周概览
3. 通过 `send-email` 发送「本月总结 + 下月预告」

### 每季度（季末最后一周）

1. 汇总本季月报
2. 写季报：核心目标 + 各维度进展
3. 检视五年计划进度
4. 通过 `send-email` 发送「本季度总结 + 下季度预告」

### 每年（年末最后一周）

1. 汇总全年季报
2. 写年报：年度核心目标 + 各季度概览 + 里程碑
3. 通过 `send-email` 发送「本年总结 + 明年预告」

## 报告写作风格

- **语气**：简洁、真诚、有温度
- **内容**：具体、可衡量，不说「学习了」而说「完成第 X 章，掌握 XX 概念」
- **润色**：保留 Zack 的原意，补充遗漏但不编造
- **隐私**：报告内容涉及个人生活，保持温暖但不过度侵入

---

## Skill 迭代规则

- 连续 3 次某章节为空 → 合并或删除
- Zack 反复提到没覆盖的维度 → 新增
- 完成率持续低于 60% → 降低目标难度
- 完成率持续高于 90% → 提高挑战度
- 每季度末回顾 skill 本身是否还贴合需求
